第6章 应用层

应用层的协议多是基于客户-服务器方式。这里的客户和服务器都是应用进程。
应用层协议规定了应用进程通信时遵循的协议。

6.1 域名系统DNS

6.1.1 域名系统概述

DNS是互联网使用的命名系统，用来把便于识别的名字转换为IP地址。

主要特点：

  • DNS是一个联机分布数据库系统，采用客户-服务器方式
  • DNS使大多数名字在本地进行解析，只有少量解析要在互联网上通信
  • 域名到IP地址的解析是由互联网上的许多域名服务器共同完成的

域名到IP地址的解析过程：

  1. 当某一应用进程需要解析域名，就调用解析程序，成为DNS的一个客户
  2. 把待解析的域名放到DNS请求报文中，以UDP用户数据报方式发给本地域名服务器
  3. 本地域名服务器查找域名后把对应的IP地址发给该应用进程
  4. 应用进程获得IP地址后即可进行通信
  5. 如果本地域名服务器不能回答该请求，就向其他域名服务器请求，此时它就成为了客户

6.1.2 互联网的域名结构

互联网采用层次树状结构的命名方法，任何一台连接在互联网上的主机或路由器都有唯一一个域名。

域名示例（四级域名）：

  • www.buaa.edu.cn ：北航主页
  • yzb.buaa.edu.cn ：北航研招办

域名的主要特点：

  • 域名由标号序列组成，最右边的标号是顶级域名，往左依次降低
  • 域名中的标号只能由英文字母、数字和 ‘-‘ 组成，不区分大小写字母
  • 完整域名不超过 255 个字符
  • 各级域名由其上一级的域名管理机构管理，顶级域名由 ICANN 管理

顶级域名(TLD)分类：

  1. 国家顶级域名：

    • cn (中国)
    • us (美国)
    • uk (英国)

  2. 通用顶级域名：

    • com (公司企业)
    • net (网络服务机构)
    • org (非营利性组织)
    • int (国际组织)
    • edu (美国教育机构)
    • gov (美国政府部门)
    • mil (美国军事部门)

  3. 基础结构域名：

    • arpa (用于反向域名解析)

我国的二级域名分类：

  1. 类别域名：

    • com (企业)
    • ac (科研机构)
    • edu, gov, mil, net, org 等

  2. 行政区域名：

    • bj (北京)等，适用于各省

6.1.3 域名服务器

域名服务器的分类：

  • 根域名服务器
  • 顶级域名服务器
  • 权限域名服务器
  • 本地域名服务器

各类域名服务器的特点：

  • 根域名服务器：

    • 最重要的服务器
    • 包含所有顶级域名服务器的域名和IP地址
    • 全球有成百上千个，分布不均衡
    • 使用了任播技术

  • 顶级域名服务器：

    • 管理在该服务器注册的二级域名

  • 权限域名服务器：

    • 负责一个区的应服务器

  • 本地域名服务器：

    • 主机查询域名时首先询问本地域名服务器
    • 计算机属性中的 DNS 服务器就是本地域名服务器

      域名解析过程的特点：

  • 主机向本地域名服务器的查询是递归查询

  • 本地域名服务器向根域名服务器的查询是迭代查询

  • 采用高速缓存来提高查询效率

6.2 文件传送协议

6.2.1 FTP概述

文件传送协议的特点：

  • FTP 使用 TCP 的可靠运输服务，采用客户-服务器模式
  • TFTP 使用 UDP 协议
  • FTP 和 TFTP 都属于复制整个文件的文件共享协议
  • 要存取文件必须先获得本地文件副本
  • 修改文件只能对副本进行修改，然后传送回原节点

6.2.2 FTP的基本工作原理

FTP 主要功能是减少或消除在不同操作系统下处理文件的不兼容性。

FTP 的服务器进程
一个 FTP 服务器进程可以同时为多个客户进程提供服务。
FTP 的服务器进程包括一个主进程和若干个从属进程：

  • 主进程：负责接受新的请求。当主进程收到客户进程发来的连接请求后，将其交给从属进程进行处理。然后回到等待状态继续接受其他客户的请求。
  • 从属进程：负责处理单个请求。

服务器进程中的主进程和从属进程是并发执行的。

文件传输功能的实现
文件传输时，FTP 的客户和服务器之间会建立两个并行的 TCP 连接：

  • 控制连接和数据连接，其中数据连接用于传输文件。因此 FTP 要使用两个端口号。
  • 两个连接对应服务器端的两个从属进程：控制进程和数据传送进程。
  • FTP 客户发送的传送请求通过控制连接发送给控制进程。然后控制进程创建数据传送进程和数据连接。

6.2.3 简单文件传送协议TFTP

简单文件传送协议 TFTP 使用 UDP 协议。

TFTP的主要特点：

  • 可用于 UDP 环境，代码简便
  • 采用了类似停止等待协议的重传机制
  • 发送后就等待确认，没有确认就重传

6.3 远程终端协议TELNET

TELNET的主要特点：

  • 远程终端协议 TELNET 采用客户服务器模型
  • 能将客户端的操作传到服务器端，然后将服务器端的输出返回到客户端屏幕
  • TELNET 采用 TCP 协议

TELNET 的服务器进程
TELNET 的服务器进程类似 FTP：

  • 由主进程等待新的请求
  • 产生从属进程来处理每一个连接

6.4 万维网www

6.4.1 万维网概述

万维网 WWW 是一个大规模的、联机式的信息储藏所。万维网的简称是 Web。

重要概念：

  • 超文本：指包含指向其他文档的链接的文本，一个超文本由多个信息源链接组成
  • 超媒体：在超文本基础上扩充包含图形、声音、视频等
  • 万维网是一个分布式的超媒体系统

Web的工作方式：

  • Web 的客户程序向互联网中的服务器程序发出请求
  • 服务器程序向客户程序送回客户所要的万维网文档
  • 页面就是在客户程序主窗口显示出的万维网窗口

Web要处理的主要问题及解决方式：

  1. 如何标志分布在整个互联网上的文档：采用统一资源定位符 URL
  2. 用什么协议来实现万维网上的链接：采用超文本传送协议 HTTP
  3. 怎么实现创作不同风格的万维网文档：使用超文本标记语言 HTML
  4. 怎样使用户很方便地找到所需信息：通过搜索引擎实现

6.4.2 统一资源定位符URL

URL的基本概念：

  • 万维网使用统一资源定位符 URL 来标志万维网上的各种文档
  • 每个文档有在互联网内唯一的 URL
  • URL 相当于指向互联网上任何可访问对象的一个指针

URL的一般形式：
<协议>://<主机>:<端口>/<路径>

  • 协议：指出采用何种协议来获取该万维网文档，一般为 http，其次为 ftp
  • 主机：即该主机的域名
  • 端口：通常都省略掉，HTTP 的默认端口号是 80
  • 路径：有时可省略

URL使用说明：

  • 输入 URL 时协议和 www 都可以省略，浏览器会自动补上
  • 使用 HTTP 的 URL 时，默认端口号是 80，通常都省略掉
  • 当路径也省略时，则 URL 指向互联网上的某个主页

主页的类型：

  1. 一个 WWW 服务器的最高级别的页面
  2. 某一个组织的一个定制的页面，从这个页面可以链接到本组织的其他站点
  3. 个人设计的 WWW 页面

URL示例：
https://www.buaa.edu.cn/jgsz1/yxsz.htm

  • cn/ 右侧的为路径
  • 最后的 .htm 表明这是一个 html 文档

6.4.3 超文本传送协议HTTP

HTTP 是面向事务的应用层协议。事务指的是一系列的不可分割的信息交换（即这些信息交换是一个整体）。

HTTP的主要特点：

  • 万维网客户与服务器程序之间交互使用的协议是 HTTP 协议
  • 万维网的客户就是浏览器
  • HTTP 本身是无连接、无状态的，使用可靠传输的 TCP 协议
    • 无连接：通信双方在交换 http 报文前不需要先建立 http 连接
    • 无状态：HTTP 服务器不记得曾经访问过的客户

HTTP连接的建立与释放：

  1. 每个万维网网点都有一个服务器进程，它不断地监听 TCP 的端口 80
  2. 当发现有浏览器向它发来 TCP 的连接建立请求并建立连接后，浏览器就会发出浏览某个页面的请求
  3. 服务器接着返回所请求的页面作为响应
  4. 最后 TCP 连接释放

‘’’
GET /dir/index.htm HTTP/1.1 //请求行，这里使用了相对 URL
Host:www.buaa.edu.cn //第一个首部行，给出了主机的域名
Connection:close //告诉服务器发送完请求的文档后就可以释放连接。
User-Agent:Mozilla/5.0 //表明用户代理使用了火狐浏览器
Accept-Launguage:cn //表明用户希望优先收到中文版本的文档
//请求报文的最后还有一个空行
‘’’

注意这个过程：

  • 首先建立 TCP 连接，且该连接的端口为 80
  • 客户会把 HTTP 请求报文作为 TCP 连接三次握手中的第三个报文的数据
  • 然后服务器直接返回文档作为响应

    HTTP/1.1：
    HTTP/1.1 协议使用了持续连接。就是万维网服务器在发送响应后一段时间内仍保持这条连接，当客户继续访问时不需要重新建立 TCP 连接。

HTTP/1.1 协议的持续连接有两种工作方式：

  • 非流水线方式：客户收到前一个响应后才能发下一个请求
  • 流水线方式：客户收到上一个响应前就可以接着发新的请求

代理服务器：

  • 代理服务器又称万维网高速缓存，它把最近的一些请求和响应暂存在本地磁盘中
  • 当新请求与暂存的请求相同，就返回暂存的响应
  • 代理服务器可以在客户端或服务端工作，也可以在中间系统上工作

代理服务器工作示例：
某个校园网使用了代理服务器：

  1. 校园网中某个主机的浏览器请求服务时，先和代理服务器建立 TCP 连接并发出 HTTP 请求报文
  2. 如果代理服务器有所请求对象就返回这个对象
  3. 如果没有，代理服务器就代表用户与源点服务器建立连接并发送 HTTP 请求报文

6.4.4 万维网的文档

万维网文档的基本特点：

  • 万维网使用超文本标记语言 HTML 来显示各种万维网页面
  • HTML 文档是一种可以使用任何文本编辑器创建的 ASCII 码文件
  • HTML 文档一般以 .html 或 .htm 为后缀
  • HTML 规定了链接的设置方法。链接一般显示为蓝色字且加上下划线，链接的终点是其他页面
  • 当链接指向的是本计算机中的文件，就是本地链接

其他相关语言：

  • 可扩展标记语言 XML 和 HTML 很相似
    • XML 用于传输数据
    • HTML 用于显示数据
  • 可扩展超文本标记语言 XHTML 是作为一种 XML 应用被重新定义的 HTML，将逐步取代 HTML
  • 层叠样式表 CSS 是一种样式表语言，用于为 HTML 文档定义布局
    • 如规定在浏览器上显示的字体、颜色、边距等

文档的类型：

  1. 静态文档
  • 文档创作完毕后就存放在万维网服务器中
  • 在用户浏览过程中，内容不会变化
  1. 动态文档
  • 文档的内容是在浏览器访问万维网服务器时由应用程序动态创建的
  • 每次访问用户看到的内容都是不一样的
  • 比如天气预报、股市行情等都要用动态文档
  • 动态文档和静态文档的差别主要在服务器一端
  • 通用网关接口 CGI 是一种标准，定义了动态文档如何创建
  • 服务器端使用 CGI 程序来创建动态文档
  • 脚本指的是被另一个程序而非处理器来解释或执行的程序。JavaScript 等就是脚本语言
  1. 活动文档
  • 可以使浏览器屏幕连续更新
  • 当浏览器请求一个活动文档，服务器直接返回一段活动文档程序的副本
  • 使该程序副本在浏览器端运行
  • 活动文档程序可以与用户直接交互，并可以连续地改变屏幕的显示

6.4.5 万维网的信息检索系统

搜索引擎的分类：

  • 全文搜索引擎（百度、谷歌等）
  • 分类目录搜索引擎（搜狐、新浪等门户网站）

全文检索搜索引擎的工作原理：

  • 通过搜索软件（如爬虫程序）到各网站搜集信息
  • 像蜘蛛爬行一样从一个网站连接到另一个网站
  • 然后建立一个在线索引数据库
  • 当用户查询时，就从已经建立的索引数据库中进行查询

Google 搜索技术的特点：

  • Google 的核心技术是 PageRank，即网页排名
  • 将搜索结果根据重要性排名
  • 关键字频率、是否知名网站等都会影响重要性

6.5 电子邮件

6.5.1 电子邮件概述

电子邮件系统的三个主要构件：

  • 用户代理（邮件客户端）
  • 邮件服务器
  • 邮件协议（包括邮件发送协议和邮件读取协议）

邮件传输协议：

  • 从用户代理到邮件服务器以及邮件服务器之间：使用 SMTP 协议
  • 用户代理从邮件服务器读取邮件：使用 POP3 或 IMAP 协议

发送邮件的过程：

  1. 用户代理使用 SMTP 协议把邮件发给”发送方邮件服务器”
  2. “发送方邮件服务器”与”接收方邮件服务器”建立 TCP 连接并发送邮件
  3. 收件人使用 POP3 协议从”接收方邮件服务器”读取邮件

邮箱地址格式： 用户名@邮件服务器的域名（如 dhb@buaa.edu.cn）

6.5.2 简单邮件传送协议SMTP

SMTP的主要特点：

  • 采用客户-服务器模式
  • 发件人的邮件存在发送方邮件服务器的邮件缓存中
  • 发送方邮件服务器定期扫描邮件缓存，发现邮件就建立连接发送
  • SMTP 的熟知端口是 25
  • 存在的问题：
    • 发送的是明文，不利于保密
    • 发送邮件不需要鉴别，易产生垃圾邮件
    • 新的 ESMTP 对这些问题进行了改进

6.5.3 电子邮件的信息格式

略。

6.5.4 邮件读取协议POP3和IMAP

常用的邮件读取协议：

  1. POP3 协议：
  • 采用客户-服务器模式
  • 功能简单有限
  • 特点是用户读取邮件后，服务器就删除该邮件
  1. IMAP4 协议：
  • 采用客户-服务器模式，但更复杂
  • 是一个联机协议
  • 用户可在本地操纵服务器邮箱
  • 只有打开邮件时才传输到用户计算机
  • 邮件在服务器保存直到用户主动删除
  • 缺点：未下载邮件时查看需要联网

6.5.5 基于万维网的电子邮件

主要特点：

  • 用户使用浏览器收发电子邮件
  • 浏览器和邮件服务器间使用 HTTP 协议
  • 邮件服务器之间仍使用 SMTP 协议
  • 无需安装用户代理软件

6.5.6 通用互联网邮件扩充MIME

MIME的主要功能：

  • 对 SMTP 进行扩充，解决其只能传送 ASCII 码的限制
  • 定义了传送非 ASCII 码的编码规则
  • 采用特殊编码方式用 ASCII 码表示其他字符
  • 指定了数百种可传送的文件类型

6.6 动态主机配置协议DHCP

DHCP的背景：

  • IP 地址中包含网络号，计算机出厂时无法预知将连接到哪个网络
  • 当计算机进入新网络时，需要配置新的 IP 地址
  • DHCP 实现了即插即用联网，无需人工配置 IP 地址

DHCP的工作原理：

  • 采用客户-服务器模式
  • 需要配置 IP 地址的主机启动时向 DHCP 服务器广播发送发现报文
  • DHCP 服务器通过提供报文分配 IP 地址
  • 可能有多个 DHCP 服务器响应，客户机选择其中一个

DHCP的主要特点：

  • 每个网络至少有一个 DHCP 中继代理（通常是路由器）
  • 分配的地址有租用期限制（可能几小时到几年不等）
  • 接近租用期时可请求更新
  • DHCP 客户的熟知端口是 68
  • DHCP 服务器的熟知端口是 67

应用示例：

  • 手机连接新的 wifi 时，需通过 DHCP 获取新的 IP 地址

6.7 简单网络管理协议SNMP

6.7.1 网络管理的基本概念

网络管理系统的组成：

  • 一个管理者
  • 多个被管设备（主机、路由器、集线器等）
  • 每个被管设备运行一个网络管理代理程序
  • 代理程序在管理程序的命令和控制下执行本地行动
  • 采用 SNMP 协议进行管理

SNMP协议的工作方式：

  • 管理程序运行 SNMP 客户程序
  • 代理程序运行 SNMP 服务器程序
  • 服务器程序持续监听客户程序的请求和命令

网络管理的基本原理：

  • 管理对象需要添加必要的软件或硬件
  • 添加的影响应尽量小
  • SNMP 最重要的思想是尽量简单

SNMP协议的三个组成部分：

  1. SNMP 本身：

    • 定义管理站和代理间交换的分组格式
    • 分组包含代理中的变量名和状态值
    • 负责读取和改变这些值

  2. 管理信息结构 SMI：

    • 定义通用规则
    • 包括对象命名、类型定义、编码规则

  3. 管理信息库 MIB：

    • 用于在被管实体中创建命名对象

6.7.2 管理信息结构SMI

SMI的三个主要功能：

  1. 被管对象的命名：

    • SMI 规定所有被管对象的名字都必须在一颗对象命名树上
    • 类似于 URL 的命名方式

  2. 被管对象的数据类型：

    • SMI 把数据类型分为两大类：简单类型和结构化类型
    • 简单类型有 Interger32 等
    • 结构化类型有 sequence（类似结构体）和 sequence of（类似数组）两种
    • SMI 采用了抽象语法记法来定义数据类型

  3. 编码方法：

    • SMI 使用基本编码规则 BER 来进行数据编码
    • BER 指明了数据类型和值
    • 把所有数据元素都组织为一个 T-L-V 三字段序列
    • T 定义数据类型，L 定义 V 字段的长度，V 定义数据的值

6.7.3 管理信息库MIB

MIB的基本概念：

  • 管理信息就是被管对象的集合
  • 被管对象必须维持供管理程序读写的若干控制和状态信息
  • 这些被管对象构成了一个虚拟的信息存储器，称为管理信息库 MIB
  • 只有 MIB 中的对象才是 SNMP 可以管理的

6.7.4 SNMP的协议数据单元和报文

SNMP的基本管理功能：

  • 读操作：用 Get 报文来检测被管对象的状况
  • 写操作：用 Set 报文来改变被管对象的状况
  • SNMP 使用无连接的 UDP

SNMP实现管理功能的方式：

  1. 使用探询操作：

    • 定期向被管设备发送探询信息
    • 了解其状况

  2. 被管对象的主动报告：

    • 代理检测到严重异常事件时
    • 主动向管理者发送报告

6.8 应用进程跨越网络的通信

6.8.1 系统调用和应用编程接口

系统调用接口的概念：

  • 系统调用接口是应用进程的控制权和操作系统的控制权进行转换的接口
  • 又称为应用编程接口 API
  • API 就是应用程序和操作系统之间的接口
  • API 和一般的函数调用很相似

套接字接口：

  • 现在的 TCP/IP 协议软件是驻留在操作系统中的
  • 套接字接口是一种供应用程序使用 TCP/IP 服务的 API
  • Windows 系统采用了套接字接口
  • 套接字是应用进程和运输层协议之间的接口
  • 是应用进程为了获得网络通信服务而与操作系统进行交互时使用的一种机制

6.8.2 几种常用的系统调用

使用TCP服务的系统调用过程：

并发服务器的工作模式：

  • 服务器要能够同时处理多个连接，即以并发方式工作
  • 采用一个主服务器进程 + 多个从属服务器进程的方式实现
  • 主服务器进程 M 不停接受新的连接请求
  • M 为每个新请求创建新的套接字，并返回标识符给客户
  • M 创建新的从属服务器进程使用新套接字与客户建立连接
  • M 继续使用原套接字接受下一个连接请求

连接建立阶段：

  • 新创建的套接字的端口号和 IP 地址都是空的
  • 服务器端应用进程调用 bind 来指明套接字的本地端口号和本地 IP 地址
  • 客户端可以调用 bind 也可以由操作系统自动分配动态端口号
  • 服务器调用 bind 后，还要调用 listen 把套接字设置为被动模式
  • UDP 服务器采用无连接方式，不使用 listen
  • 服务器调用 accept 来完成分配从属服务器进程与新套接字
  • 客户端创建套接字后，调用 connect 向服务器发出连接请求

数据传送阶段：

  • 客户和服务器都调用 send 来传送数据，调用 recv 来接收数据
  • send 调用需要三个变量：
    • 数据要发往的套接字的描述符
    • 要发送的数据的地址
    • 数据的长度
  • recv 调用需要三个变量：
    • 要使用的套接字的描述符
    • 缓存的地址
    • 缓存空间的长度

连接释放阶段：

  • 调用 close 来释放连接和撤销套接字

6.9 P2P应用

P2P 应用采用了 P2P 体系结构：没有固定的服务器，绝大多数交互都使用对等方式。
目前 P2P 工作方式下的文件共享占据了互联网流量中最大的份额，比万维网所占比例大得多。

6.9.1 具有集中目录服务器的P2P工作方式

第一代P2P文件共享特点：

  • 所有用户机上文件的地址存放在一台服务器上
  • 用户下载资源时先向该服务器询问资源地址
  • 再从存储资源的计算机下载资源

6.9.2 具有全分布式结构的P2P文件共享程序

BT下载的主要特点：

  • BT 采用”最稀有的优先”的技术
  • 参与某个文件分发的所有对等方构成一个”洪流”
  • 每个洪流都有一个”追踪器”
  • 对等方加入洪流时要向追踪器登记
  • 优先向传送速率最高的相邻对等方传送文件块

6.9.3 P2P文件分发的分析

当对等方的数量很大时，采用 P2P 下载文件比传统的客户-服务器模型快很多。

6.9.4 在P2P对等方中搜索对象

分布式散列表DHT：

  • P2P 中广泛使用的索引和查找技术
  • 是一个分布式数据库
  • 数据库包含两部分信息：
    • 关键字是资源名
    • 值是存放对象的节点的 IP 地址

Chord算法：

  • 是一种基于 DHT 的算法
  • 采用散列函数将资源名映射为均匀分布的数字（标识符）
  • 将标识符放到 Chord 环上
  • 保存资源名的主机也通过散列函数映射到环上作为节点
  • 每个资源名保存到环上离它最近的节点所对应的主机中
